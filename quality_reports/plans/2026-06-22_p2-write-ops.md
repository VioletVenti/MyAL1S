## Context（为什么做这个）

P0/P1 把读链路（课表/作业/成绩/公告/材料/回放）+ agent 对话 + 持久化全部打通。但 pku3b 唯一的写原语 `submit_file`（`pku3b/src/api/blackboard.rs:1084`）一直**刻意未暴露**（`tools.rs:11` 注释 + 测试断言）。P2 要把这扇门打开，但**安全地**打开：用户配权限矩阵（默认「需确认」），agent / UI 触发写时过闸，写不可逆所以默认要人确认。这是 roadmap 里 P2「写操作 + 权限矩阵」的第一个竖切——只做交作业，把整条「写工具 → 闸 → 矩阵 → 确认 → 执行 → 审计」架构跑通并验证，P3 的选课/发帖复用同一套闸。

## Locked decisions（grilling 已锁定，不再推翻）

1. **写范围 = 仅交作业**（复用 `submit_file`）。
2. **双通道**：UI 与 Agent 共用同一闸 + 矩阵。
3. Agent 通道 Active（能写作业；树洞 P3）。
4. **计划1（两阶段、REST、不引 WebSocket）**：agent 调写工具 → 工具体**创建 pending 审批**并返回 `{status:"pending_approval"}` → **agent.run 正常结束**（不用 pydantic-ai 原生 deferred-run resume，那套重启会丢、序列化脆，留作未来增强）→ 用户在「待审批」面板确认 → 后端在 run **之外**经 `gateway.call_tool` 执行写。
5. 矩阵 **两级 deny / confirm**；`auto` 留位（枚举槽 + 闸里 `else` 分支，P3 点亮），P2 显式 no-op，不静默别名到 confirm。
6. 文件来源 = **聊天附件，先传后写**：ChatBox 上传 → 后端存 → 得 `file_id` → agent 带 `file_id`（绝不带路径）调写工具。
7. **UI 直交 = 隐式确认**（用户点了按钮+选了文件=已确认）→ 立即执行，但仍过闸查 deny；写一条 `executed` 审批行作审计。
8. 矩阵粒度 = **语义组**（写工具属 `assignment_submission` 组；矩阵按组设级）。

## 架构（新增 Seam 7 + uploads 工具）

```
            ┌─────────── UI 直交（隐式确认）──────────┐    ┌─── Agent（计划1）──┐
作业行「交作业」→ 上传文件→file_id                     用户聊天附文件→file_id
            │  POST /api/submit {assignment_id,file_id}   "把这个交到作业X"
            │                                              │ (backend 把 file_id 注入消息)
            │                                              ▼
            │                              agent.run → LLM 调本地 submit_assignment(assignment_id,file_id)
            │                                              │ 工具过闸:create_approval → 返回 pending
            │                                              ▼ agent.run 结束
            │                                              ▼
            ▼                          POST /api/approvals/{id}/decide  ← 用户在「待审批」面板点确认/拒绝
   ┌────────────────────────────────────────────────────────┐
   │  PermissionGate (Seam 7, backend/app/permissions.py)    │
   │   level_for(group) · create_approval · decide · execute_now │
   │   ── 单一私有 _execute(approval) ── 两通道都走这里 ──┐  │
   │   deny→拒; confirm→pending/执行; auto→P3 no-op       │  │
   └──────────────────────────────────────────────────────┼──┘
                                    │ resolve file_id→绝对路径 (uploads.py)
                                    ▼ gateway.call_tool("submit_assignment", {assignment_id, file_path})
                          pku3b MCP `submit_assignment`（path 原语，read_only:false，
                          对 agent 不可见；经 direct_call_tool 调用）
                                    ▼ submit_file(&Path) → Blackboard
```

**关键不变量（延续架构主线）**：写工具的**执行**仍是确定性的 `gateway.call_tool`（不过 LLM）；agent 只负责「请求」写。读链路（composer/dashboard/deterministic）保持**无副作用**——新增结构测试禁止它们 import `permissions.py`。

**两通道共享单一 `_execute`**：`decide()`（agent 确认后）和 `execute_now()`（UI 隐式确认）都调同一个私有 `_execute(approval) -> 信封`，里面做 file_id→路径解析 + `gateway.call_tool` 派发。这是「双通道共用一闸」的本质——派发到 Blackboard 的代码只有这一处。

**Seam 7 = PermissionGate** 是 Composer 的写侧对偶：都持 `(store, gateway)`，Composer 做读侧 join，Gate 做写侧 gate+派发。同 altitude。**不并入 Store**（Store 是哑 SQL，不管 live 数据）；**不并入 Composer**（Composer 无副作用，结构测试禁 `pydantic_ai`）。`uploads.py` 是 2 方法的 I/O 工具（`save`/`path_for`），**不是 seam**——从 Gate 拆出避免策略耦合文件布局。

**对 agent 隐藏 path 原语**（安全必须）：agent 工具集 = `FilteredToolset(self._server, filter_func=...name!="submit_assignment")` + 本地 `FunctionToolset` 写工具 `submit_assignment(assignment_id, file_id)`。LLM 永远看不到带 `file_path` 参数的工具（防路径注入）。`self._server`（裸）仍归 `gateway.call_tool`/`direct_call_tool` 用——过滤只改 agent 可见性，不改派发。（`FilteredToolset` 已在 pydantic-ai 1.107 验证：`toolsets/filtered.py`。）

### Increment A — pku3b 写原语 + id 重解析  `[RISK: 最高]`
- **依赖**：无。**文件**：`pku3b/src/mcp/tools.rs`（+ 参考 `api/blackboard.rs:1084` `submit_file`、`:953-958` `handle.id()` 哈希）。

- **抽取共享爬虫 helper**：把 `list_assignments`（`tools.rs:311-380`）的 `login_blackboard→get_courses(true)→content_stream drain→build_content().into_assignment_opt()` 抽成私有 `walk_assignments()`，`list_assignments` 与新工具共用——避免两份 walk 漂移导致哈希对不上。新工具：walk → 匹配 `handle.id()==assignment_id`（哈希不可逆，必须重走+匹配）→ `assignment.submit_file(Path::new(&file_path))`；无匹配返回 `error` 信封。
- 改 `tools.rs:9-12` 模块 doc：`submit_file` 不再「刻意不暴露」，改为「作为 `submit_assignment` 暴露，由消费侧 P2 矩阵 gate」。
- **修两处破测**：① `catalog_named_and_login_is_only_non_read_only`（`tools.rs:708-729`）——「login 是唯一非只读」改为「login + submit_assignment 非只读」，删 `!contains("submit_assignment")`。② `test_list_tools_exposes_read_only_catalog`（`backend/tests/test_mcp_gateway.py:39-46`）——原断言 `not any("submit" in n)` 会立刻破；改为断言 `submit_assignment` **在**裸 `list_tools()` 且 `readOnly=false`（「对 agent 隐藏」那半推迟到 D）。

### Increment D — Agent 写工具 + 隐藏 path 原语  `[RISK: 最高]`
- **依赖**：A+B+C（闭包 gate）。**文件**：`backend/app/mcp_gateway.py`、`backend/app/main.py`、`backend/app/routes/chat.py`、`backend/tests/test_mcp_gateway.py`。
- **解决构造顺序结**（最高风险）：agent 在 `McpGateway.__init__`（`mcp_gateway.py:92`）建，但 `store`/`gate` 在 `main.py:32` lifespan 才存在。加 `gateway.attach_write_toolset(gate)`，由 lifespan 在 gate 建好后调：构造 `FunctionToolset`（闭包 `gate`）、`FilteredToolset(self._server, filter_func=lambda c,td: td.name!="submit_assignment")`、重建 `self._agent = Agent(..., toolsets=[filtered_server, write_ts])`。`self._server`（裸）留给 `direct_call_tool`。
- 本地工具 `submit_assignment(assignment_id, file_id)`：调 `gate.create_approval(group="assignment_submission", ...)` → 返回 `{status:"pending_approval", approval_id, summary}`。**工具自身不调 `gateway.call_tool`**——只有 gate 派发。
- **附件交接**（`chat.py`）：ChatBox 发 `attachment_file_id` + 消息；`chat.py` 在 `agent.run` 前把 `（已上传附件：{filename}，file_id={id}）` 注入用户消息文本，使 LLM 能填 `file_id`。（保持附件不进 MCP catalog。）
- 改 `SYSTEM_PROMPT`（`mcp_gateway.py:54-67`）：**line 66「当前为只读阶段」现已不实**——改为：agent 可经工具**请求**交作业，工具返回 pending 由用户在「待审批」面板确认；**同一作业确认结果出来前不要重试**；提交状态以 `list_assignments`/「待审批」面板为准，**勿凭记忆断言已交**。
- **测试（三段密闭断言）**：agent 可见工具不含任何带 `file_path`/`path` 参数的 submit；裸 `_server.list_tools()` **仍含** `submit_assignment`（证明是过滤而非删除）；`gateway.call_tool("submit_assignment",…)` **仍可达**；调 agent 工具→建审批行+返回 pending。A 增量推迟的「隐藏」半断言在此落地。
- **风险**：构造顺序（agent 早于 store）+ 过滤密闭性（泄漏=LLM 见服务器本地路径）+ run-ends 语义三碰头。`FilteredToolset` API 已验证，风险在集成。

## Risks & open items

- **最高风险**：A 的 id 重解析 walk（必须与 `list_assignments` 字节一致）；D 的构造顺序结（agent 早于 store）+ 过滤密闭性。
- **执行时 OTP**：session 可能在请求与确认间过期→`_execute` 返 `needs_otp`→`failed`+重登 CTA（已覆盖）。
- **agent 如何在后续轮知道已执行**：采用 (a) 重查 `list_assignments` + (c) 系统提示令其 defer 到「待审批」面板，**勿凭记忆断言**。合成消息注入（b）留 P3（需 `ModelMessagesTypeAdapter` 往返测试）。
- **并发**：`transition_approval` 用 `WHERE status='pending'` 守卫防双 decide。
- **run.sh**：若加 `uploads_dir` 环境变量，加进 preflight 免得一键启动默默写到错误 CWD。
- **auto 级别**：P2 不实现（no-op），P3 文件无关写（树洞/选课）点亮 `else` 分支。
